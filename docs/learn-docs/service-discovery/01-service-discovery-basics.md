# 服务注册与发现基础概念

## 1. 什么是服务注册与发现？

在微服务架构中，服务实例是动态变化的（IP 地址、端口、实例数量），服务注册与发现机制解决了**服务间如何互相找到对方**的问题。

### 核心角色

```
┌─────────────┐      注册       ┌─────────────────┐
│   服务提供者   │ ────────────► │   注册中心        │
│  (Provider)  │               │  (Registry)      │
└─────────────┘               └─────────────────┘
                                     ▲
┌─────────────┐      发现/订阅        │
│   服务消费者   │ ────────────────────┘
│  (Consumer)  │
└─────────────┘
```

| 角色 | 职责 |
|------|------|
| **注册中心** | 维护服务实例列表，健康检查 |
| **服务提供者** | 启动时注册，定期发送心跳 |
| **服务消费者** | 订阅服务列表，负载均衡调用 |

## 2. 核心流程

### 2.1 服务注册

```
Provider 启动
    ↓
向 Registry 发送注册请求
    ↓
Registry 存储实例信息（IP、端口、元数据）
    ↓
定期发送心跳（默认 5s）
    ↓
超时未心跳 → Registry 标记为不健康
超过阈值 → 剔除实例
```

### 2.2 服务发现

```
Consumer 启动
    ↓
向 Registry 订阅目标服务
    ↓
获取实例列表（拉取 + 推送）
    ↓
本地缓存实例列表
    ↓
调用时：负载均衡选择实例
```

### 2.3 健康检查

| 方式 | 说明 |
|------|------|
| **心跳模式** | Provider 定期发送心跳（Nacos、Eureka） |
| **主动探测** | Registry 主动检查 Provider（Zookeeper） |

## 3. 主流注册中心对比

| 特性 | Nacos | Eureka | Consul | Zookeeper |
|------|-------|--------|--------|-----------|
| **CAP** | AP/CP 可切换 | AP | CP | CP |
| **健康检查** | 心跳 + TCP/HTTP | 心跳 | 多种 | Keep-Alive |
| **配置中心** | ✅ 内置 | ❌ | ✅ | ❌ |
| **命名空间** | ✅ | ❌ | ✅ | ❌ |
| **Spring Cloud** | ✅ | ✅ (停更) | ✅ | ✅ |
| **社区活跃度** | 高 | 低 | 中 | 中 |

## 4. 为什么选择 Nacos？

### 优势

1. **功能全面**：注册中心 + 配置中心二合一
2. **AP/CP 切换**：临时实例（AP）vs 持久实例（CP）
3. **命名空间隔离**：支持多环境（dev/test/prod）
4. **权重路由**：支持流量灰度
5. **国内生态好**：阿里开源，中文文档完善

### 适用场景

- 微服务架构的服务治理
- 动态配置管理
- DNS 服务发现

## 5. 关键概念

### 5.1 临时实例 vs 持久实例

| 类型 | 健康检查 | 适用场景 |
|------|---------|---------|
| **临时实例** | 心跳（AP 模式） | 无状态服务，大多数场景 |
| **持久实例** | 服务端主动探测（CP 模式） | 有状态服务，需强一致性 |

### 5.2 命名空间（Namespace）

用于**环境隔离**或**租户隔离**：
- `dev` / `test` / `prod`
- 不同租户的服务互不可见

### 5.3 分组（Group）

用于**业务隔离**：
- 默认分组：`DEFAULT_GROUP`
- 可按业务线分组：`ORDER_GROUP`、`PRODUCT_GROUP`
