# 01 - SDD 方法论：规范驱动开发

> 本文回答：SDD 到底是什么？它反转了什么？为什么偏偏是现在可行？核心原则有哪些？
> 主要来源：Spec Kit 仓库根目录的 `spec-driven.md`（SDD 方法论深度说明）。

## 1. 一句话理解 SDD

**Spec-Driven Development（SDD）**：把"规范（specification）"从"代码的附属文档"提升为"开发源头"，让规范足够精确、完整、无歧义，从而**直接生成**可工作的实现，而不是仅仅"指导"实现。

> 关键词是 **executable**（可执行）。规范不是写给人看的旁注，而是能产出代码的"源"。

## 2. 权力反转：代码为王 → 规范为王

这是 SDD 最核心的哲学断言，理解它就理解了 SDD 的一切设计动机。

### 传统模式：代码是真理

几十年来，**代码是源、文档是附属**：

- 写 PRD 是为了"指导"开发，写完就被搁置
- 设计文档是"参考"，代码一旦写出来，文档很快过时
- 代码是 single source of truth，文档最多是"良好的愿望"
- 代码往前走，规范很少跟上 → 意图与实现之间的 gap 一直存在

传统方法试图用"更好的文档""更详细的需求""更严格的流程"来**缩小**这个 gap，但都默认 gap 是不可避免的。

### SDD 模式：规范是真理

SDD **反转（inverts）** 这个权力结构：

- 规范不服务代码，**代码服务规范**
- PRD 不是实现的指南，而是**生成实现的源头**
- 技术方案不是" informing 编码"的文档，而是**产出代码的精确定义**
- 当规范和实现计划能生成代码时，**gap 不再被缩小，而是被消除**——只有"转换"

> 用原文的话：*"There is no gap—only transformation."*

### 这个反转带来的连锁后果

一旦规范成为源头，整个开发工作流都重新组织：

| 传统概念 | SDD 下的含义 |
|---------|-------------|
| 维护软件 | 演进**规范**，而不是改代码 |
| 调试 | 修复**生成错误代码的规范/实现计划** |
| 重构 | 为清晰度重组**规范** |
| 加新功能 | 回到规范，创建新的实现计划 |
| 并行实现 | 从同一规范派生多个实现（不同技术栈/优化目标） |

开发者的注意力从"机械翻译规范到代码"解放出来，聚焦到**创造力、实验、批判性思考**。

## 3. SDD 工作流概览（实践视角）

方法论在实践中的样子（命令细节见 02 篇）：

```text
想法（模糊）
  ↓ 与 AI 对话迭代
PRD（完整规范）
  ↓ AI 生成实现计划，研究代理收集技术上下文
实现计划（每个技术决策有理由、可追溯到需求）
  ↓ 规范稳定后生成代码
代码（领域概念→数据模型，用户故事→API，验收场景→测试）
  ↓ 生产反馈回流
规范演进（性能瓶颈→新非功能需求，安全漏洞→新约束）
```

几个值得注意的特点：

- **需求和设计变成连续活动**，而不是离散阶段
- **团队流程友好**：规范被 review、版本化、在分支里创建与合并
- **一致性校验是持续的**，不是一次性 gate——AI 持续分析规范的歧义、矛盾、缺口
- **测试场景是规范的一部分**，不是代码写完后再补——规范同时生成实现和测试

## 4. 核心原则（6 条）

SDD 的设计围绕这六条原则，Spec Kit 的每个机制都能映射回它们：

### 4.1 Specifications as the Lingua Franca（规范是通用语言）

规范成为主要产物，代码是它在特定语言/框架下的表达。**维护软件 = 演进规范**。

> 开发的"通用语言"上移到更高层次，代码是"最后一公里"。

### 4.2 Executable Specifications（可执行规范）

规范必须**精确、完整、无歧义**到能生成可工作系统。这消除了意图与实现之间的 gap。

- 这不是"写得更详细"那么简单，而是要求规范达到"能生成代码"的精度
- 模板里的 `[NEEDS CLARIFICATION]` 标记就是逼你把模糊处显式化（见 03 篇）

### 4.3 Continuous Refinement（持续精炼）

一致性校验**持续进行**，不是一次性门禁。AI 持续分析规范的歧义、矛盾、缺口。

- 对应 Spec Kit 的 `/speckit.analyze`、`/speckit.clarify`、`/speckit.converge` 命令

### 4.4 Research-Driven Context（研究驱动的上下文）

研究代理在整个规范过程中收集关键上下文：库兼容性、性能基准、安全影响、组织约束。

- 对应 `plan.md` 阶段产出的 `research.md`（见 02 篇）

### 4.5 Bidirectional Feedback（双向反馈）

生产现实反过来塑造规范演进。指标、事故、运维学习都成为规范精炼的输入。

- 性能瓶颈 → 新的非功能需求
- 安全漏洞 → 影响所有未来生成的约束

### 4.6 Branching for Exploration（分支化探索）

从同一规范生成多个实现方案，探索不同优化目标：性能、可维护性、用户体验、成本。

- 这就是"并行实现"，Spec Kit 用 Git 分支承载不同 feature 规范

## 5. 为什么是现在（三大趋势）

SDD 这个想法并不新（规范驱动早就有），但原文论证了**为什么偏偏现在变得既可行又必要**：

### 趋势一：AI 能力到达临界点

自然语言规范现在能**可靠地**生成可工作代码。这不是取代开发者，而是**放大**他们的效能——自动化"规范到实现"的机械翻译。

> 还能放大探索与创造力，支持"推倒重来"、增删改、批判性思考。

### 趋势二：软件复杂度指数增长

现代系统集成几十个服务、框架、依赖。用人工流程让所有部分与原始意图对齐越来越难。SDD 用**规范驱动的系统性对齐**来应对。

- 一个推论：未来框架可能演化为"AI-first 支持"而非"human-first 支持"

### 趋势三：变化节奏加速

需求变化比以往都快，pivot 不再是例外而是常态。传统开发把变化当"扰动"——每次 pivot 都要手动穿透文档、设计、代码，要么慢要么欠债。

SDD 把需求变化变成**正常工作流**：

- 改 PRD 里的核心需求 → 受影响的实现计划自动更新
- 改用户故事 → 对应 API 端点重新生成
- pivot 变成**系统性再生成**，而不是手动重写

> 还能做 what-if/simulation：如果要改造应用去卖更多 T 恤，会怎么实现、怎么实验？

## 6. 开发阶段（SDD 适用的三类场景）

Spec Kit 把 SDD 的适用场景分成三个阶段，工具设计兼容全部：

| 阶段 | 焦点 | 关键活动 |
|------|------|---------|
| **0→1 开发**（Greenfield） | 从零生成 | 高层需求 → 规范 → 实现计划 → 生产级应用 |
| **创造性探索** | 并行实现 | 探索多样方案、多技术栈/架构、实验 UX 模式 |
| **迭代增强**（Brownfield） | 存量现代化 | 迭代加功能、现代化遗留系统、适配流程 |

> 对存量项目，Spec Kit 建议**把工具更新和规范产物演进分开**：升级时刷新受管项目文件，行为变更时更新 `specs/` 产物。具体见 `docs/guides/evolving-specs.md`。

## 7. 实现途径（不绑定工具）

原文强调 SDD 是**方法论**，今天可以这样实践（不一定要用 Spec Kit）：

- AI 助手做迭代式规范开发
- 研究代理收集技术上下文
- 代码生成工具把规范翻译为实现
- 适配规范优先工作流的版本控制
- 通过 AI 分析规范文档做一致性检查

> 关键是：**把规范当源头，代码是服务规范的生成产物**。Spec Kit 只是把这套做法工程化、工具化了。

## 8. SDD vs 传统开发：一个对比直觉

spec-driven.md 给了一个很直观的"聊天功能"对比：

**传统方式**（约 12 小时文档工作）：

```text
1. 写 PRD（2-3h）→ 2. 设计文档（2-3h）→ 3. 手动搭项目结构（0.5h）
→ 4. 技术规范（3-4h）→ 5. 测试计划（2h）
```

**SDD 方式**（约 15 分钟）：

```bash
/speckit.specify Real-time chat system with message history and user presence
/speckit.plan WebSocket for real-time messaging, PostgreSQL for history, Redis for presence
/speckit.tasks
```

15 分钟后你有：完整的用户故事与验收标准、带技术选型理由的实现计划、API 契约与数据模型、测试场景、全部在 feature 分支里版本化。

> 注意：这个对比不是"省时间"这么简单。**真正的价值在于结构化强制的一致性与完整性**——模板保证不漏、决策可追溯、文档与代码同源。时间数字是营销性的，理解机制比记数字重要。

## 9. 规范持久化模型（三个层面）

Martin Fowler 的 SDD 工具综述把规范的"生命周期"分三层，Spec Kit 把它作为概念引入（见 `docs/concepts/spec-persistence.md`）：

| 模型 | 含义 | 适用 |
|------|------|------|
| **Spec-first** | 写规范后就丢弃，只为启动开发 | 一次性 spike |
| **Spec-anchored** | 实现后保留规范，用于后续改动 | 常规项目 |
| **Spec-as-source** | 规范是唯一人写源，实现产物从它再生成 | 规范即契约 |

Spec Kit 还区分了第二个问题：**需求变化时，产物集怎么变？** 三个模型（见 `docs/concepts/spec-persistence.md`）：

- **Flow-back spec**：任何产物都可改，事后手动对齐。快，但易静默漂移。
- **Flow-forward spec**：完成的产物不可变，新需求开新 feature 目录。可审计，但有重复。
- **Living spec**：`spec.md` 是契约，`plan.md`/`tasks.md` 是派生。改 spec 先，再重生成下游。

> 这是**团队约定**，不是 CLI 设置。Spec Kit 不强制任何一个，留给团队选。这种"提供工作流但不规定产物维护策略"的克制设计值得注意。

## 10. 小结

- SDD 反转了"代码为王"，让**规范成为源头、代码成为派生**
- 这之所以现在可行，是因为 AI 到了临界点、复杂度要求系统性对齐、变化节奏要求 pivot 常态化
- 六条核心原则中，"可执行规范"和"持续精炼"是工程上最硬的要求
- SDD 是方法论，Spec Kit 是它的工具化落地——下一篇开始看怎么用
