## 一、没有 CI 的世界

要理解 CI 的价值，先体会一下没有 CI 的痛。

### 1.1 一个真实的场景

假设一个 5 人团队在开发 my-mall 商城项目，没有 CI，开发流程是这样的：

1. **各自本地开发**：每个人在自己分支上写功能，mall-order 的人改了下单接口，mall-product 的人改了商品查询接口。
2. **合并代码前**：想起来要跑一遍测试——「算了，就改了两行，应该没问题」，直接合并（测试经常被跳过）。
3. **合并到主干后**：部署到测试环境——启动失败。原来 mall-order 调用 mall-product 的新接口，但 mall-product 还没合并那部分代码。互相阻塞，排查半天。
4. **生产部署时**：手动 `mvn package` 打 jar 包，手动 `scp` 传到服务器，手动重启——漏传了一个 `application-prod.yml` 配置文件，线上起不来。
5. **凌晨 2 点上线出 bug**：要回滚，得手动翻聊天记录找「上一个能用的版本是哪个」，再手动重新部署。全员熬夜。

### 1.2 没有 CI 的三大问题

把上面的场景抽象一下，核心就是三个问题：

| 问题 | 表现 | 后果 |
|------|------|------|
| **集成太晚** | 问题在合并时才暴露，甚至到生产环境才暴露 | 排查成本高，因为「坏掉」时已经混入了大量改动，难以定位是哪次提交引入的 |
| **手动易错** | 构建、测试、部署全靠人手动执行 | 会忘（不跑测试）、会漏（漏传配置）、会错（打错分支的包），人是不靠谱的执行者 |
| **反馈太慢** | 从提交代码到知道「是否 OK」，可能要几小时甚至几天 | 开发者早就切到下一个功能了，回来修 bug 时上下文已经丢失 |

这三个问题不是个例，而是**所有靠人工保证质量的团队都会遇到的系统性问题**。CI 就是为了解决它们而生的。

## 二、CI 是什么

### 2.1 定义

**持续集成（Continuous Integration，CI）** 的核心思想是：

> **频繁地将代码集成到主干，每次集成都通过自动化构建和测试来验证。**

拆解三个关键词：

| 关键词 | 含义 | 反面（不这么做会怎样） |
|--------|------|----------------------|
| **频繁** | 每天多次提交、多次集成，不是攒一周再合并 | 攒一周再合并 → 冲突巨大、问题堆积、难以定位 |
| **自动化** | 构建和测试由机器执行，不靠人手动跑 | 靠人手动跑 → 会忘、会偷懒、会跳过 |
| **验证** | 每次提交都得到明确的「能跑 / 不能跑」反馈 | 没有反馈 → 不知道这次提交是否破坏了什么，直到出事 |

### 2.2 一个类比：安检门

CI 就像地铁站的**安检门**：

- 你每过一次安检门（每次提交代码），都要接受检查（自动构建 + 测试）。
- 有问题立刻报警（CI 失败），你当场处理。
- 而不是攒一周的行李，周日一次性过安检——那时候问题已经堆积成山，而且你根本不知道是哪件行李里有违禁品。

换句话说，CI 把「质量检查」从**一次性、滞后、人工**的大检查，变成**高频、即时、自动**的小检查。问题在产生的瞬间就被发现，而不是积攒到爆炸。

### 2.3 CI 的本质

CI 的本质不是某个工具（不是 Jenkins、不是 GitHub Actions），而是一种**工程实践**：

- 工具只是实现手段
- 核心是「**频繁集成 + 自动验证 + 快速反馈**」这套理念
- 换成 Jenkins、GitLab CI、CircleCI，理念完全一样，只是语法不同

> 💡 这也是为什么本章不讲 GitHub Actions 语法——语法是工具层面，会换；CI 的理念是工程层面，不会变。先把理念搞懂，再学语法就只是「查文档套模板」的事。

## 三、CI / CD 的关系

经常看到 CI/CD 连写，这三个概念到底是什么关系？

### 3.1 三个概念对照

| 缩写 | 全称 | 做什么 | 触发 | 产出 |
|------|------|--------|------|------|
| **CI** | Continuous Integration 持续集成 | 自动构建 + 测试 | 每次 push / PR | 「代码能跑」的信心 |
| **CD** | Continuous Delivery 持续交付 | CI 通过后自动打包产物 | CI 通过 | 随时可部署的产物（镜像 / jar） |
| **CD** | Continuous Deployment 持续部署 | 产物自动部署到生产 | 产物就绪 | 运行中的线上服务 |

> 注意：两个 CD 缩写相同，含义不同。一个停在「产物就绪」（交付），一个走到「线上运行」（部署）。区别就在于**部署这一步是否自动**。

### 3.2 三者关系图

```text
开发者提交代码
    │
    ▼
┌─────────┐     ┌─────────┐     ┌─────────┐
│   CI    │────▶│   CD    │────▶│   CD    │
│ 构建测试 │     │ 打包产物 │     │ 自动部署 │
│         │     │（镜像）  │     │（K8s）  │
└─────────┘     └─────────┘     └─────────┘
 持续集成         持续交付         持续部署
```

### 3.3 几点说明

- **三者界线看团队**：有的团队 CI 通过就自动部署到测试环境（CI + 持续部署到 staging），生产部署需手动点按钮（持续交付到 prod）。通常连写 CI/CD，不严格区分。
- **CI 是基础**：没有 CI 谈 CD 是空中楼阁——你连「代码能不能跑」都不知道，自动部署上去的只会是坏的代码。
- **本项目本次设计范围** = CI 全链路 + 持续交付（镜像发布到 Harbor）。持续部署（ArgoCD 自动部署 K8s）属于后续阶段。详见项目设计文档 [ci-cd/overview.md](../../standards/ci-cd/overview.md) 第三章的边界划分。

> 🎯 对 my-mall 而言：CI 阶段产出「测试通过的代码」，镜像发布阶段产出「Harbor 中的镜像 tag」，ArgoCD 阶段消费镜像 tag 部署 K8s。三者通过**镜像 tag** 这一契约解耦——CI 不关心怎么部署，CD 不关心怎么构建。

## 四、CI 流水线长什么样

### 4.1 流水线的概念

CI 的执行过程叫**流水线（Pipeline）**。一条流水线由多个**阶段（Stage）**组成，代码从提交到通过，像工厂流水线一样经过各道工序：

```text
代码提交 → 构建 → 测试 → 质量检查 → 产物打包
  │         │       │        │          │
  ▼         ▼       ▼        ▼          ▼
 checkout  mvn    surefire  jacoco    docker
           verify  failsafe  覆盖率    build
```

每一道工序（stage）失败了，流水线就停下来报警——不会继续往后走。这保证了**只有通过所有检查的代码才能产出制品**。

### 4.2 各阶段详解

| 阶段 | 目的 | 本项目的实现 | 失败了怎么办 |
|------|------|------------|------------|
| **构建** | 编译代码，确认能编译通过 | `mvn verify`（含编译，14 个 Maven 模块 reactor 构建） | 修复编译错误 |
| **测试** | 跑单元 / 切片 / 集成测试 | surefire（单元/切片）+ failsafe（集成） | 修复测试用例或被测代码 |
| **质量检查** | 覆盖率门禁、lint、类型检查 | jacoco 覆盖率门禁（后端）+ `pnpm lint` / `type-check`（前端） | 补测试 / 修 lint / 修类型错误 |
| **产物打包** | 构建镜像 / jar | `docker build` + 推送 Harbor（仅 main / tag） | 检查 Dockerfile、构建上下文 |

### 4.3 快速失败原则

注意上面流水线的顺序不是随便排的。一个好的 CI 流水线遵循**快速失败（Fail Fast）**原则：

- 把**最快能出结果**的检查放最前面（lint、编译），最慢的放后面（集成测试、构建镜像）。
- 这样编译都过不了的话，几秒钟就反馈，不用等几分钟跑完测试才发现。
- 典型顺序：`lint → 编译 → 单元测试 → 集成测试 → 质量门禁 → 构建产物`。

> 类比：体检时先量血压（快），再做胃镜（慢）。血压都不正常就没必要排队做胃镜了，先回去调理。CI 也是——编译都不过，跑集成测试就是浪费时间。

## 五、CI 的核心价值

用一张表总结 CI 带来的价值，每条配上「没有 CI 时」的对比，感受会更直观：

| 价值 | 没有 CI 时 | 有 CI 后 |
|------|----------|---------|
| **尽早发现问题** | 合并时 / 生产时才暴露，问题已经混入大量改动 | 提交后几分钟内自动反馈，问题在产生的瞬间被发现 |
| **减少手动操作** | 手动构建、手动测试、手动部署，全靠人 | 全自动，人只负责写代码和 Code Review |
| **保证主干质量** | 主干可能随时是坏的，没人敢随便拉代码 | 主干永远通过测试，随时可部署、可拉取 |
| **快速回滚** | 出问题手动翻记录找「上一个能用的版本」 | 每次 commit 有对应镜像 tag，精确回滚到任意版本 |
| **团队协作** | 合并冲突频繁、互相阻塞、「我这里能跑啊」 | PR 自动检查，CI 通过才能合并，合并即安全 |

其中最核心的是第一条「尽早发现问题」。软件工程里有一条铁律：**bug 发现得越晚，修复成本越高**。在提交时发现（CI 阶段），改两行就行；到生产才发现，可能要回滚、赔用户、写事故报告。CI 的价值就是把 bug 的发现时机**尽可能提前**。

## 六、CI 的关键实践

CI 不是「装个工具就完事」，要真正发挥作用，需要遵循一套工程实践。这些不是某个工具的语法，而是理念层面的原则。

### 6.1 频繁提交，小步合并

- 每天多次提交，每次提交小改动。
- 不要攒一周再合并——攒得越久，冲突越大、问题越多、定位越难。
- 小步合并的意义：每次合并的改动小，万一出问题，排查范围也小。

> 类比：吃饭要细嚼慢咽，不要攒一周一次吞一大盆——那叫「噎死」不是「吃饭」。代码集成也是，小口吃才好消化。

### 6.2 快速反馈

CI 流水线要**快**，目标是在 5-10 分钟内出结果。为什么快很重要？

- 开发者提交后会等 CI 结果——超过 10 分钟，注意力就转移了，回头修 bug 时上下文已丢失。
- 慢的 CI 会被开发者「绕过」——「等它跑完太久了，我先合并吧」，CI 形同虚设。

让 CI 快的常用手段：

| 手段 | 说明 | 本项目应用 |
|------|------|----------|
| 快速失败 | 最快的检查放最前面 | lint → 编译 → 测试 → 构建 |
| 缓存依赖 | 缓存 Maven 仓库、node_modules，避免每次重新下载 | composite action 缓存 `~/.m2` 和 pnpm store |
| 并行执行 | 无依赖的 job 并行跑 | 后端 CI 和前端 CI 并行（取两者较长者，而非之和） |
| Docker layer 缓存 | 复用未变更的镜像层 | 多阶段 Dockerfile + buildx 缓存 |

### 6.3 自动化一切

- 构建、测试、质量检查全部自动化，不依赖任何人工步骤。
- 唯一需要人的环节：**Code Review**——这是机器替代不了的判断（架构合理性、业务逻辑正确性）。
- 反面：如果 CI 里有一步是「人工确认 xxx」，那就不算真正的自动化，迟早会成为瓶颈。

### 6.4 主干永远可部署

- CI 通过的代码才能合并到主干。
- 通过**分支保护规则**强制：PR 必须等 CI 通过（required status check）才能 merge，不能手动绕过。
- 效果：主干（main / develop）永远处于「可部署」状态，随时能拉来上线，不用提心吊胆。

> 这是 CI 最理想的终态：**主干永远是绿的**。任何时刻有人说「现在上线」，你都能从容应对，因为你知道主干一定是通过测试的。

### 6.5 一次构建，到处使用

- CI 产出的镜像 / artifact 直接用于部署，**不要在部署时重新构建**。
- 为什么？如果在部署时重新构建，构建出来的产物可能和 CI 验证的不一致（依赖版本漂移、本地环境差异），那 CI 的验证就白做了。
- 本项目的体现：CI 阶段构建镜像推到 Harbor，ArgoCD 直接拉 Harbor 镜像部署，**不会重新编译**。CI 验证过的镜像 = 部署的镜像，一字不差。

## 七、CI 触发策略

CI 不是「任何时候都全跑一遍」，要讲究**什么时候触发、跑哪些东西**。这叫触发策略。

### 7.1 触发时机

| 触发时机 | 本项目策略 | 目的 |
|---------|-----------|------|
| Pull Request | 跑 CI（不发布镜像） | 合并前检查，卡门禁——CI 没过不让合并 |
| Push 到 develop | 跑 CI（不发布镜像） | 开发主线持续集成 |
| Push 到 main | 跑 CI + 镜像发布 | 稳定分支，合并即交付（持续交付） |
| Push tag `v*.*.*` | 跑 CI + 镜像发布 + Release | 正式发版 |

### 7.2 为什么 feature 分支不构建镜像？

这是个值得想一想的决策。feature 分支的 PR 只跑 CI（构建 + 测试），**不构建镜像、不推 Harbor**。原因：

- **避免镜像仓库膨胀**：每个 feature 分支 PR 都产镜像，Harbor 会快速堆积大量垃圾镜像（feature 分支可能从不合并）。
- **PR 阶段只需要验证「代码能跑」**：镜像构建是「交付」动作，PR 还没合并，交付为时过早。
- **镜像只在「确定要上线」时构建**：main 合并（要交付）或 tag（要发版）才构建，每次构建的镜像都是有意义的。

> 这是 CI 和 CD 边界的体现：CI（构建 + 测试）在 PR 就跑，CD（打包产物）只在合并到主干后跑。**质量验证可以高频，产物交付要克制**。

## 八、CI 流水线的设计思路

讲清设计 CI 时的思考框架，为本项目方案做铺垫。这些是设计层面的选择题，不同项目答案不同。

### 8.1 单文件 vs 可复用组合

CI 工作流怎么组织？两种主流方式：

| 方式 | 特点 | 适用 |
|------|------|------|
| **单文件** | 一个 `.github/workflows/ci.yml` 承载所有逻辑，用 `if` 条件区分阶段 | 简单但臃肿，适合小项目（1-2 个 job） |
| **可复用组合** | 按职责拆成多个子工作流，编排器串联调用；公共步骤抽成 composite action | 模块化、可复用、可分阶段落地，适合多模块项目 |

本项目选了**可复用组合**：14 个后端模块 + 1 个前端，单文件会臃肿到 300+ 行难以维护。拆成 `backend-ci` / `frontend-ci` / `docker-publish` / `release` 子工作流，由编排器 `ci.yml` 统一调度。详细对比见项目设计文档 [ci-cd/overview.md](../../standards/ci-cd/overview.md) 第二章。

### 8.2 串行 vs 并行

Job 之间什么时候串行、什么时候并行？

- **有依赖关系的串行**：发布依赖 CI 通过（`docker-publish` needs `backend-ci` + `frontend-ci`）。
- **无依赖的并行**：后端 CI 和前端 CI 互不依赖，并行跑，总耗时 = 两者较长者，而非之和。

```text
         ┌── backend-ci ──┐
detect ──┤                ├── docker-publish ── release
         └── frontend-ci ─┘
   (并行)              (串行，依赖上面通过)
```

### 8.3 全量 vs 增量

- **全量构建**：每次构建所有模块，准确但慢。
- **增量构建**：只构建变更的模块，快但有误判风险（模块间有传递依赖时可能漏构建）。
- **本项目第一版选全量**：14 个模块用 Maven reactor 一次 `mvn verify` 全构建，准确为先。后期可用 `-T 1C`（每核一个线程）优化，暂不上增量，避免过度优化。

### 8.4 paths 过滤

不是所有变更都值得跑 CI：

| 变更路径 | 触发 | 不触发 |
|---------|------|--------|
| `mall-*/**`、`pom.xml`（后端代码） | 后端 CI | 前端 CI |
| `mall-admin-frontend/**`（前端代码） | 前端 CI | 后端 CI |
| `docs/**`、`*.md`（文档） | 都不触发 | 后端 + 前端 CI |

> 核心价值：改一行前端 CSS，不该触发 14 个模块的 Maven 构建（耗时数分钟）；改文档不该触发任何 CI。这是 paths 过滤的意义——**让 CI 资源花在刀刃上**。

## 九、CI 与分支策略的关系

CI 的触发策略不是凭空定的，要和 **Git 分支策略**对齐。分支策略定义了「代码怎么流动」，CI 触发策略定义了「代码流动时跑什么检查」——两者必须配套，否则会出现「该检查的没检查」或「不该跑的浪费资源」。

本项目分支策略（详见 [docs/git-workflow.md](../../standards/git-workflow.md)）与 CI 行为的对应关系：

| 分支 | CI 行为 | 说明 |
|------|--------|------|
| `feature/*` | PR 时跑 CI | 功能分支，通过 PR 合并到 develop |
| `develop` | push 时跑 CI | 开发主线，持续集成 |
| `main` | push 时跑 CI + 发布镜像 | 稳定分支，合并即交付 |
| `hotfix/*` | PR 时跑 CI | 紧急修复，通过 PR 合并到 main |
| tag `v*.*.*` | 全链路 CI + 发布镜像 + Release | 正式发版 |

> 💡 关键点：**CI 的门禁作用通过分支保护规则落地**。光有 CI 不够，还得在 GitHub 仓库设置里把「CI 通过」设为合并的必需条件（required status check），否则开发者可以无视 CI 失败直接合并，CI 就形同虚设。分支保护 = CI 门禁的「执法权」。

## 十、小结与下一步

### 10.1 核心知识点回顾

| 知识点 | 一句话总结 |
|--------|----------|
| CI 解决的问题 | 集成太晚、手动易错、反馈太慢 |
| CI 的定义 | 频繁集成 + 自动化构建测试 + 快速验证反馈 |
| CI 流水线 | 构建 → 测试 → 质量检查 → 产物打包，遵循快速失败原则 |
| CI 的核心实践 | 频繁提交、快速反馈、自动化一切、主干可部署、一次构建到处使用 |
| 触发策略 | PR 跑 CI 卡门禁，main 合并发布镜像，tag 发版，与分支策略对齐 |
| CI/CD 关系 | CI 是基础（验证代码），CD 交付产物（镜像），持续部署上线（K8s） |

### 10.2 记住一个类比

把 CI 想成**安检门**：

- 频繁过安检（频繁提交），而不是攒一周过一次。
- 安检自动化（机器检查），不靠人自检。
- 有问题立刻报警（快速反馈），当场处理。

这三句话就是 CI 的全部精髓。
