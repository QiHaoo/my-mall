# 配置中心基础

## 1. 什么是配置中心？

在微服务架构中，配置中心是**集中管理所有服务配置**的组件，解决配置分散、难以维护的问题。

### 传统配置方式的问题

```
服务A ──► application.yml（本地文件）
服务B ──► application.yml（本地文件）
服务C ──► application.yml（本地文件）
...
```

**痛点**：
- 配置分散在几十个服务中
- 修改配置需要重启服务
- 环境切换需要维护多套文件
- 配置变更无法追溯

### 配置中心方案

```
              ┌───────────────────┐
              │    配置中心        │
              │  (Nacos/Apollo)   │
              └─────────┬─────────┘
                        │
        ┌───────────────┼───────────────┐
        │               │               │
   ┌────┴────┐     ┌────┴────┐     ┌────┴────┐
   │  服务A   │     │  服务B   │     │  服务C   │
   └─────────┘     └─────────┘     └─────────┘
```

**优势**：
- 配置集中管理
- 动态刷新，无需重启
- 环境隔离
- 版本控制 + 回滚

## 2. 配置 vs 数据

| 类型 | 特点 | 存储 |
|------|------|------|
| **配置** | 低频变更、启动时加载 | 配置中心 |
| **数据** | 高频变更、运行时读写 | 数据库/缓存 |

**配置示例**：
- 数据库连接串
- 业务开关（促销开关、灰度开关）
- 阈值（限流阈值、超时时间）
- 第三方服务地址

## 3. 核心概念

### 3.1 Data ID

配置的唯一标识，通常与服务名关联：

```
mall-coupon.yaml      # 服务默认配置
mall-coupon-dev.yaml  # 开发环境配置
mall-coupon-prod.yaml # 生产环境配置
```

### 3.2 Group

配置的逻辑分组：

```
DEFAULT_GROUP  # 默认分组
TEST_GROUP     # 测试环境
PROD_GROUP     # 生产环境
```

### 3.3 Namespace

物理隔离，用于环境或租户隔离：

```
my-mall    # 开发环境
my-mall-test  # 测试环境
my-mall-prod  # 生产环境
```

### 3.4 配置优先级

```
Namespace > Group > Data ID
```

同一配置在不同 Namespace/Group 中可以有不同的值。

## 4. 主流配置中心对比

| 特性 | Nacos | Apollo | Spring Cloud Config |
|------|-------|--------|---------------------|
| **部署复杂度** | 低 | 高 | 中 |
| **实时推送** | ✅ | ✅ | ❌ (需 Bus) |
| **版本管理** | ✅ | ✅ | ✅ (Git) |
| **灰度发布** | ✅ | ✅ | ❌ |
| **配置回滚** | ✅ | ✅ | ✅ (Git) |
| **权限管理** | 基础 | 完善 | 基础 |
| **中文文档** | 好 | 好 | 一般 |

## 5. 配置中心架构

```
┌─────────────────────────────────────────┐
│            配置中心服务端                  │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐  │
│  │ 配置存储 │  │ 推送服务 │  │ API 服务 │  │
│  └─────────┘  └─────────┘  └─────────┘  │
└─────────────────────────────────────────┘
                    │
        ┌───────────┼───────────┐
        │           │           │
   ┌────┴────┐ ┌────┴────┐ ┌────┴────┐
   │  Client │ │  Client │ │  Client │
   │ (服务A) │ │ (服务B) │ │ (服务C) │
   └─────────┘ └─────────┘ └─────────┘
```

### 工作流程

1. **启动时**：Client 从 Server 拉取配置
2. **运行时**：Server 推送配置变更到 Client
3. **变更时**：管理员在控制台修改配置
