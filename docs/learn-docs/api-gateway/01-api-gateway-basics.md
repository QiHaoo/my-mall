# API 网关基础

## 1. 为什么需要 API 网关

### 1.1 没有网关的问题

```
客户端 ──► 服务A (端口 7000)
        ├──► 服务B (端口 7100)
        ├──► 服务C (端口 7200)
        └──► 服务D (端口 7300)
```

- 客户端需要知道所有服务的地址和端口
- 每个服务都要独立做鉴权、限流
- 跨域问题每个服务都要处理
- 服务拆分或合并时，客户端代码要改

### 1.2 引入网关后

```
客户端 ──► 网关 (端口 1000) ──► 服务A
                              ├──► 服务B
                              ├──► 服务C
                              └──► 服务D
```

- 客户端只需知道网关地址
- 鉴权、限流、日志等横切关注点统一在网关处理
- 服务拆分对客户端透明

## 2. 网关核心职责

| 职责 | 说明 |
|------|------|
| **路由转发** | 根据请求路径/Header 转发到对应服务 |
| **负载均衡** | 在多实例间分发请求 |
| **统一鉴权** | 校验 Token、权限，未登录直接拦截 |
| **限流熔断** | 保护下游服务不被流量打垮 |
| **跨域处理** | 统一处理 CORS |
| **请求日志** | 记录所有请求的出入参 |
| **协议转换** | HTTP → gRPC、HTTP → WebSocket |
| **灰度发布** | 按 Header/IP/权重 将流量路由到新版本 |

## 3. 反向代理 vs API 网关

```
反向代理（Nginx）           API 网关（Spring Cloud Gateway）
     │                              │
     │  流量入口                     │  流量入口
     │  负载均衡                     │  负载均衡
     │  SSL 终止                    │  SSL 终止
     │  静态资源                     │  + 动态路由（基于服务发现）
     │  缓存                        │  + 业务鉴权（JWT/OAuth2）
     │  限流（简单）                 │  + 业务限流（按用户/接口）
     │                              │  + 请求改写、协议转换
     │                              │  + 熔断降级
```

**本质区别**：
- **反向代理**：偏基础设施，配置静态，不感知业务
- **API 网关**：偏应用层，配置动态（可从 Nacos 拉取），感知业务逻辑

**实际部署**：Nginx（外层） + Spring Cloud Gateway（应用层） 双层部署

```
客户端 → Nginx (SSL终止/静态资源) → Spring Cloud Gateway (路由/鉴权/限流) → 微服务
```

## 4. 主流网关方案对比

| 特性 | Spring Cloud Gateway | Zuul 2.x | Kong | APISIX |
|------|---------------------|----------|------|--------|
| 语言 | Java | Java | Lua/Nginx | Lua/Nginx |
| 模型 | 响应式（WebFlux） | 异步非阻塞 | Nginx + Lua | Nginx + Lua |
| 生态 | Spring Cloud 深度集成 | Netflix 生态 | 独立部署 | Apache 开源 |
| 配置 | YAML / 代码 / Nacos | 代码 / 配置中心 | Admin API | etcd |
| 性能 | 高（Netty） | 中 | 极高 | 极高 |
| 适用 | Spring Cloud 项目 | Netflix 项目 | 多语言项目 | 多语言高性能项目 |

**本项目选择 Spring Cloud Gateway**：
- 与 Spring Cloud Alibaba 生态无缝集成
- 支持从 Nacos 动态拉取路由配置
- 与 Sentinel 集成实现限流
- 团队技术栈统一（Java）

## 5. BFF 模式

BFF（Backend For Frontend）：为不同客户端提供专属网关

```
移动端 App ──► Mobile BFF ──► 微服务集群
Web 前端   ──► Web BFF    ──► 微服务集群
第三方     ──► Open BFF   ──► 微服务集群
```

**优势**：
- 不同客户端的数据格式、字段裁剪可以不同
- 移动端可以聚合多个微服务接口，减少请求次数
- 各端独立演进，互不影响

**本项目**：当前使用单一网关，后续如有多端需求可拆分 BFF。
