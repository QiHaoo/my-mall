属性分组管理

```
获取分类的属性分组列表：路径参数携带三级分类id，并可以携带查询参数（一个关键词，然后可以对多个字段进行同时搜索）以及分页参数（这里需要common提供分页相关的工具类）、排序。还有支持查所有，我看别人的实现是三级分类id传0就是查所有，这是他们的方案，不一定适合我们，我们按生产级最佳实践来设计。

新增属性分组

获取属性分组详情：响应里面要额外提供所属分类的path(参数类型是数组，一级分类id/二级分类id/三级分类id，注意这里path的获取要在分类那边相关接口中获取，目前那边还没有，实现的时候要补充)，方便前端展示。

修改属性分组

删除属性分组

与属性关联关系
    获取当前分组关联的所有属性列表
    批量删除关联关系
    移除关联关系（复用批量删除接口）
    获取当前分组可以关联的所有属性列表（分组所属分类下没有被其他分组关联的属性）
    批量新增关联关系（分组id、属性id）


```

规格参数

```
新增规格参数：属性图标、所属分类必选，所属分组非必选（后端要根据选择的分类、分组，建立对应的关联关系）（这里属性-分组关系，虽然有个多对多的关系，但是业务上好像其实一个属性只会归属于一个分组）
获取分类下的规格参数列表：分页、排序（desc/asc）、关键词（响应中要额外返回分组名称和分类名称，方便前端展示）(支持获取所有)
获取规格参数详情：包含分组id、分类id、分类完整路径（数组类型，一级分类id/二级分类id/三级分类id）、分组名称、分类名称
修改规格参数：分组关系的处理，要判断是否需要新增关系还是修改关系
批量删除规格参数
```

销售属性
```
获取分类下的销售属性列表：分页、排序、关键词（支持获取所有）（和规格参数列表接口一样，只是类型不一样，在规格参数列表接口中加个type参数（看是否要放到路径参数）来区分即可，不需要新增接口）
新增销售属性：同样不需要新增接口，新增页面把类型固定为销售属性，然后不可修改即可。不需要维护分组关系，销售属性不需要跟分组关联。
```

直接访问管理前端，然后在登录的时候又会通过网关访问（网关暂时把所有请求路由给管理后端，使用路径重写filter，把/api开头（前端配置了baseUrl为/api）的请求重写成/admin），需要在网关中处理跨域（与此同时要关掉管理后端的跨域配置，不然跨域响应头的内容会出现多个，比如allow-origin头的值会出现两个域地址，一个是网关加的，一个是管理后台加的）
