# common 模块文档组织规范

> common 模块（`mall-common`）是基础设施模块，不是业务模块，文档组织方式与业务模块不同。

---

## 一、为什么不套用业务模块的"overview + 功能域"模式

业务模块（如 mall-product）按功能域拆文档，因为功能域有独立的表、Controller、Service，内容量大、读者场景是"开发某功能 → 查对应文档"。

common 模块不同：

| 维度 | 业务模块 | common 模块 |
|------|---------|-------------|
| 组织方式 | 按功能域划分 | 按组件/能力划分 |
| 组件关系 | 有依赖链（分类→SPU→SKU） | 组件相对独立 |
| 读者场景 | "开发分类管理" → 查功能域文档 | "统一响应怎么用" → 查组件说明 |
| 单组件复杂度 | 高（表/接口/DTO/错误码） | 多数低（1~2 个类） |
| 迭代频率 | 高（跟着业务走） | 低（基础设施稳定） |

逐组件出文档会导致文件爆炸且每篇只有几十行，一篇文档带速查表反而最高效。

---

## 二、文档组织方式

### 2.1 主文档：一篇 `common-module-design.md`

common 模块维护一篇主文档 `common-module-design.md`，涵盖：

- 包结构总览
- 自动装配机制
- 各组件职责（开头带**组件速查表**，可快速定位）
- 依赖管理
- 与规范文档的关系
- 演进记录

### 2.2 复杂组件：独立设计文档

当某个组件复杂到需要独立的架构设计（如 OSS SDK 涉及 MinIO 架构、Presigned URL 流程、前端直传），拆出独立设计文档：

| 独立文档 | 内容 | common 主文档的职责 |
|---------|------|-------------------|
| `mall-product/object-storage-design.md` | 对象存储完整设计 | §3.10 只留"做什么 + 怎么装配 + 链接到详细设计" |

### 2.3 新增组件时的处理

| 组件复杂度 | 处理方式 |
|-----------|---------|
| 小组件（1~2 个类，如 PageQuery、UserContext） | 直接加到 `common-module-design.md` 的"各组件职责"章节 + 速查表 |
| 复杂组件（涉及架构设计、多方协作，如分布式锁 SDK、限流组件） | 独立设计文档 + common 主文档留索引 |

---

## 三、与 AGENTS.md 模块文档规范的关系

AGENTS.md 中的"模块文档组织规范"主要面向业务模块（overview + 功能域）。common 模块遵循该规范的**精神**（分层、按需演进、模板一致），但不强制套用 overview + 功能域的结构，而是采用本文档定义的"主文档 + 复杂组件独立文档"模式。
