# 文档写作规范

> 本文档定义项目开发文档的写作规范，包括模块文档组织方式、图表选型规则。
> AGENTS.md 仅保留索引指针，详细规范统一在此维护。

---

## 一、模块文档组织规范（按需演进 + 分层结构）

### 1.1 文档分层

每个服务模块的文档分两层：

| 层级 | 文件 | 职责 |
|------|------|------|
| 模块概述 | `overview.md` | 讲全局：模块职责、功能域划分、核心概念、整体数据模型、上下游关系 |
| 功能域文档 | `{feature}.md` | 讲具体：需求、接口、DTO、错误码 |

功能域文档之间通过相对链接互相引用，不重复内容。

### 1.2 按需演进，不强制前置规划

文档跟着开发节奏走，开发到哪写到哪：

- 不要求一开始就写完整的 `overview.md`——可以先写功能域文档，模块内容积累到一定程度再抽取共性形成概述
- 概念性内容暂时无处安放时，先放当前功能域文档，等 `overview.md` 出现时再迁移

### 1.3 粒度控制

一个功能域一篇文档，不把多个功能域塞进同一篇。

**判断标准**：如果一块功能有独立的表、独立的 Controller、独立的 Service，就值得单独一篇文档。参考 `docs/mall-product/` 下的分类管理、品牌管理。

### 1.4 模板一致

功能域文档统一遵循 `category-management.md` / `brand-management.md` 的章节结构：

```
一、业务背景
二、功能需求（功能清单 + 业务规则）
三、前端设计（可选，涉及前端页面时）
四、数据模型
五、接口设计
六、DTO / VO 定义
七、错误码
八、网关路由
九、HTTP 调试文件
十、非功能性要求
十一、实现清单
```

> **章节顺序设计**：采用「需求 → 前端设计 → 后端实现」的阅读路径。先讲功能是什么、用户看到什么（前端页面），再讲后端如何支撑（数据模型、接口、DTO），最后是实现清单。这符合全栈单人开发的认知顺序：理解需求 → 明确页面 → 设计接口 → 落地实现。

### 1.4.1 前端设计章节规范

当功能域涉及前端页面时，在"功能需求"之后、"数据模型"之前增加"前端设计"章节（即第三章）。纯后端功能（如定时任务、消息消费）无需此章节，后续章节顺延。

**内容范围**（只讲"页面结构和交互"，不重复接口定义）：

```markdown
## 三、前端设计

### 页面布局
> Figma 设计稿：[链接]()
>
> Figma 出稿前以文字描述作为页面结构契约。

### 组件拆分
| 组件 | 职责 | 消费接口 |
|------|------|---------|
| CategoryTree | 三级分类树展示+拖拽排序 | GET /product/category/tree |
| CategoryForm | 新增/编辑表单 | POST/PUT /product/category |

### 关键交互
- 拖拽排序：el-tree + draggable，拖拽后调用 PUT /product/category/sort
- 批量删除：多选确认弹窗，调用 DELETE /product/category/batch
```

**原则**：
- 不重复接口定义（引用本文档"五、接口设计"章节）
- 不写详细的前端实现逻辑（放前端代码注释）
- 只写页面结构、组件职责、关键交互与 API 的对应关系
- 组件拆分表的"消费接口"列只写接口路径（如 `GET /product/category/tree`），读者可跳转第五章查看细节

**Figma 设计稿要求**：
- 每个功能域的**关键页面**必须出 Figma 设计稿，文档「页面布局」小节贴 Figma 链接
- Figma 稿可由 AI Agent 直接生成（通过 figma-console-mcp，详见 `docs/frontend/figma-mcp-guide.md`），也可由设计师/开发者手动完成
- AI 生成时：先写文字描述作为页面结构契约，再调用 MCP 工具生成 Figma 稿，最后将链接补回文档
- 手动出稿时：先在文档以文字描述作为占位契约，Figma 稿产出后将链接补回
- 文字描述保留作为快速浏览概要

### 1.4.2 前端设计独立文档拆分规则

前端设计默认内嵌在功能域文档的第三章。当功能域前端复杂度较高时，拆分到独立文档 `docs/frontend/{module}/{feature}.md`。

**拆分触发条件**（满足任一即拆分）：
- 前端设计章节超过 150 行
- 前端有独立的设计决策需要详细记录（如动态表单生成、复杂状态管理、多组件联动）
- 功能域文档因前端内容过长导致后端接口设计被稀释

**拆分后关系**：功能域文档第三章保留简短摘要（页面布局概述 + 组件拆分表 + 链接到独立文档），详细设计放独立文档。

**当前状态**：分类管理、品牌管理内嵌在功能域文档，未拆分。SPU/SKU 等复杂模块按需拆分。

### 1.5 overview.md 索引职责

概述文档维护该模块所有功能域文档的链接索引，读者从概述入口能快速定位到任意功能域。

## 二、文档图表规范（Mermaid vs 纯文本）

### 2.1 用 Mermaid 的场景

流程图 / 架构图（flowchart）、时序图（sequenceDiagram）、状态图（stateDiagram）、ER 图（erDiagram）、类图（classDiagram）、Git 分支图（gitGraph）、甘特图（gantt）、思维导图（mindmap）—— 即有明确节点关系、流向、状态流转的图，**一律用 Mermaid，禁用 ASCII 画线框图**。

### 2.2 用纯文本的场景

目录树（`tree` 风格缩进）、层级嵌套结构（如包结构、DTO 列表）、配置项清单、简单的层级缩进说明 —— 这些用文本缩进更紧凑直观，不要硬套 Mermaid。

### 2.3 Mermaid 语法要求

使用 ` ```mermaid ` 代码块（注意语言名是 `mermaid` 全小写），Material 主题自动渲染，支持深浅色切换。

### 2.4 选择原则

能表达清楚的前提下，选更简洁的方式。简单层级用文本，复杂关系用 Mermaid。
